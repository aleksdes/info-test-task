## Установка

1. Клонируйте репозиторий на локальную машину.
2. Установите необходимые зависимости с помощью команды:
   ```
   pnpm install
   ```

3. Запустите проект с помощью команды:
   ```
   pnpm dev
   ```

## Процесс разработки

### Git
На проекте используется Git flow, три основные ветки
- Main - продакшн
- Develop - ветка разработки
- Stage - превью продакшена

Общие правила и они важные
- Постарайся избегать ребазирования (rebase) он переписывает историю, можно потерять код как свой, так и твоего тиммейта
- Постарайся чаще коммитить и заливать коммиты на сервер, чаще, как можно чаще
- В заголовке (на первой строке) коммита пиши простым языком, например для бага пиши что-то вроде `bugfix(JIRA код задачи): Не работал компонент выбора даты`.
  Остальные строки в твоем распоряжении, помни что заголовки коммита часто используются для написания release note. 
  Написание release note по заголовкам сильно облегчает работу лиду и менеджеру,
- Не делай сквош, коммиты надо хранить для истории работ и исправлений
- cherry-pick нельзя, только в крайних случаях и только после согласования с командой 

> Важно! В ветку main запрещены пуш коммитов напрямую, только через реквест и только hotfix

Эти три ветки основные, прямой пуш в них ЗАПРЕЩЕН, за исключением хотфиксов.

#### Разработка фичи
Для разработки фичи надо отколоться от ветки `develop` в ветку `feature/<JIRA код задачи>`
Всю разработку нового функционала надо проводить только в фича ветках.
Фича ветки отправляются на ревью с запросом на слияние в `develop`, то есть `develop <- feature/...`

#### Ветки исправления багов
Для исправления багов доступны только ветки `develop` и `stage`
Смотря на какой ветке обнаружился баг, с той же ветки откалываемся в ветку `bugfix/<JIRA код задачи>`,
По готовности создается pull/merge request в ту же ветку с которой откололись, то есть `develop` или в `stage`

А что если баг на проде? Исправление багов на продакшн доезжают аналогичным маршрутом что и фичи и багфиксы, за исключением 
критических багов, которые порождают аварию и панику(~~ядра~~) заказчика, в случае критической аварии нужно отколоться от `main`
в ветку `hotfix/<JIRA код задачи>` и направить pull request в ветку `main`

Твой код должен путешествовать по веткам только так
```
main <- stage <- develop <- feature/someFeature 
```

### Архитектура

Приложение как пирог разделено на слои, для корректного разделения кода, сначала разбей задачу на слои.
Давай на примере корзины разберем флоу, нужно сделать корзину, путь будет такой
- Корзина что-то хранит, а хранилище это сущность, значит нужен `entities/cart/` срез, пишем в него весь бизнес корзины
  по части работы со стором, и компоненты, например `CartItemRow.vue`.
- Корзина как то с пользователем взаимодействует, то что взаимодействует это фича, нужен `features/cart/`, туда пишем 
  бизнес в части добавления в корзину, удаления, короче все мутации, то что сущность изменяет. Сюда же и компоненты, например,
  `AddToCart.vue`, `ClearCart.vue`
- А корзина то это виджет же? Представим что корзина в выпадашке отображается, а там кнопки удаления и очистки корзины, значит
  скомпонуем его в `widgets/cart`, там опишем финальную корзину и просто объединим компоненты из `features/cart`

Но мысли в контексте конкретной фичи, например в корзине может быть форма оформления заказа, да, это как бы часть корзины, но нет,
это отдельная итерация пользователя, он сначала воспользовался корзиной, в потом пошел заказ оформлять, формы оформления заказа и все прочее,
это уже отдельна фича и соответственно отдельная ответственность, повторяем тот же путь только для модуля `Order`

Мысли в контексте итераций пользователя а не в контексте программы, первое время это будет сложно, потом очень просто.
Разделяй взаимодействия пользователя по фичам, добавить в Корзину, удалить из Корзины - это будет частью корзины и только,
а вот, заказать ИЗ корзины - это будет частью итерации заказа корзины.

### Иерархия слоев

Важно соблюдать иерархию основных слоев, правило простое,
верхний слой не может импортировать слой ниже,
слой ниже может импортировать из слоя выше

> features слой может использовать наборы компонентов, композиций и вызывать бизнес код из слоя entities,
> но никак иначе, смотри иерархию

```
|-- entities
|   |-- features
|   |   |-- widgets
```

### Правила экспорта
Каждый срез слоя, (например features/feature1, entities/user) должен содержать index.ts файл с реэскпортом публичного api.
Эскпорт звездочки запрещен, это нарушает принцип ответственности среза, срез предоставляет только публичный API
который могут использовать другие слои в иерархии, нельзя экспортировать все, экспортируй только результат

> Например из слоя entities/user можно экспортировать модель, стор Pinia но нельзя экспортировать различные утилиты, и вспомогательный код 
> который используется только в entities/user

Это плохо
```ts
export * from './something'
```
Это хорошо 
```ts
export { useUserStore, type UserEntity } from './model/'
```
