# Слой `pages` — правила и паттерны по текущей кодовой базе

Документ фиксирует **реально применённые** в `src/pages` паттерны и даёт рекомендации для единообразия нового кода. Базовые определения слоя `pages` в FSD — в скилле `applying-fsd-architecture`.

---

## Роль слоя `pages` в этом проекте

В текущей реализации страницы — это **композиционный корень экрана** для `<router-view>`:

- собирают **виджеты** (`@/widgets/...`), **фичи** (`@/features/...`), **общий UI** (`@/shared/ui`, PrimeVue при необходимости);
- подключают **сущности** (`@/entitties/...`) — сторы/композаблы домена, часто через `useAsyncOperation`, `storeToRefs`;
- задают **разметку страницы** (часто с `HeaderPage`), локальные отступы и **scoped**-стили;
- в ряде экранов держат **сквозную оркестрацию**: SSE-подписки, синхронизацию фильтров дат, обработчики ухода со страницы.

Отдельных сегментов **`api/`**, **`model/`**, **`ui/`** внутри `pages` **нет** но она **допустима в рамках FSD**: вся логика живёт в `<script setup>` рядом с шаблоном.

---

## Организация каталогов

По верхнему уровню слой разбит **не строго “одна папка = один маршрут”**, а по **зоне продукта / доступа**:

| Каталог   | Назначение |
|-----------|------------|
| `auth/`   | экраны входа и OAuth-callback (Keycloak и legacy login). |
| `user/`   | основная зона пользователя (дашборд, расписание, сотрудники, FAQ, полезные ссылки и т.д.). |
| `admin/`  | админские экраны (роли, FAQ по id, загрузка ITPH). |
| `<название зоны>/`  | иные зоны в рамках требований проекта. |
| `errors/` | страница ошибок (сейчас — `404`). |

**Рекомендация для новых экранов**: класть страницу в ту зону (`user` / `admin` / `auth` / `<название зоны>`), к которой относится маршрут и политика доступа в роутере; имя подпапки — **`kebab-case`**, как у существующих слайсов (`useful-links`, `faq-id`, `itph-upload`).

## Публичный API страницы (`index.ts`)

Почти у каждой страницы есть **`index.ts`** в папке слайса со страницей:

- паттерн: `export { default as <ИмяЭкспорта> } from './<Файл>.vue'`;
- имя экспорта совпадает с тем, что импортирует **`src/app/router-setup/routes/*`** (например `StatisticsPage`, `SchedulePage`, `FaqIdPage`).

**Исключение**: `errors/404.vue` подключается из роутера **прямым импортом файла** (`import _404Page from '@/pages/errors/404.vue'`), без `index.ts`.  
**Рекомендация**: для новых страниц предпочитать вариант **с `index.ts`**, чтобы импорт из приложения оставался единообразным (`@/pages/...` без `.vue`).

### Баррель по зоне `auth/`

`src/pages/auth/index.ts` реэкспортирует подпапки:

- `export * from './login'`
- `export * from './login-keycloak'`
- `export * from './callback-keycloak'`

Роутер импортирует `{ Login, LoginCallback, LoginKeyClock } from '@/pages/auth'`.

Такое же поведение может быть и для других зон в `src/pages/<название зоны>`. В зоне должен находиться `index.ts`, который реэкспортирует подпапки

## Зависимости: что типично импортируют страницы

Снизу вверх по FSD (как в коде):

- **`widgets/`** — крупные блоки экрана (дашборд, расписание, FAQ и т.д.).
- **`features/`** — сценарные куски (селектор ресторана, скачивание, поиск FAQ, debug-форма на логине).
- **`shared/ui`**, **`shared/lib`** — `HeaderPage`, `FilterByDate`, `AsyncWrapper`, `useAsyncOperation`, `dayjs`, `parseDate`, SSE-клиент, типы из `shared/generated/api`.
- **`entities/`** — доменные сторы и хуки

**Паттерн загрузки данных на странице**: обёртка запроса в `useAsyncOperation` + отображение состояний через `AsyncWrapper`

## Композиция layout на странице

Повторяющийся каркас:

- корневой контейнер с классом в духе **BEM**: `statistics-page`, `schedule-page` и т.п.;
- **`HeaderPage`** с `title`, часто `sticky`, слоты `#append` / `#append-outer` под селектор ресторана и фильтры по дате;
- для страниц с таблицами/формами — **`SelectorRestaurantFeature`** и др. фичи в хедере или над контентом.

Стили:

- **`scoped`** + **`lang="scss"`**;
- вложенность по БЭМ: `&__content`, `&__header`;
- переопределение внешнего вида хедера через **CSS-переменные** (`--header-page--padding`, `--header-page--top` и т.д.);

## Связь с роутингом и навигацией

### Импорты из `app` (роут-хелперы и guards)

Страницы **напрямую** импортируют:

- фабрики маршрутов / навигацию: `useFaqRoute`, `useDashboardRoute`, `useLoginRoute` из **`@/app/router-setup/routes/*.ts`**;
- общий обработчик предупреждения о потере данных, но их может и не быть в проекте вообще: **`dataLossWarningHandler`** из `@/app/router-setup/router-guards/data-loss-warning-handler.ts`.

Примеры: `FaqPage.vue`, `FaqIdPage.vue`, `errors/404.vue`, `LoginCallback.vue`, страницы расписаний с несохранёнными изменениями.

**(Расхождение с каноническим FSD.)** В классической схеме слой **`pages`** не должен зависеть от слоя **`app`** (роутер и приложение — выше по графу). Здесь зависимость **от `app/router-setup` допущена как практический контракт проекта** (типизированные `use*Route` и общие guards рядом с декларацией маршрутов).
**Важно** - Этот механизм НИКОГДА НЕ НАДО переписывать ни в ходе рефакторингов, ни в ходе разработки нового функционала. Новый код писать с учетом этих особенностей зависимости в слое `page` с `app`

### Логика навигации внутри страницы

На нескольких экранах (`SchedulePage`, `users-schedule-calendar`, `AupScheduleCalendar`) в `<script setup>` регистрируется **`router.beforeEach`** для предупреждения о несохранённых данных + **`beforeunload`**.

**(Расхождение / компромисс.)** Такая логика ближе к **сценарию ухода со страницы** и по FSD чаще ожидается в **`features`/`widgets`/`model`**, а страница только подключает готовый композабл. В кодовой базе это **сосредоточено в странице** — при добавлении похожих экранов либо копируют этот паттерн для единообразия, либо постепенно выносят в общий composable.

## Тяжёлая логика на странице (SSE, таймеры, агрегация)

На страницах встречается:

- **`useSSESubscriber`** + `watch` на параметры подписки + `patch` в сторы сущностей;
- интервалы и `watch` на ресторан/пользователя (например локальное время для таймлайна);
- прямая работа с **`useRouter`**, `useRoute`, `useTemplateRef` к виджетам.

Это отражает роль страницы как **клея между виджетами и глобальным состоянием**.  
**Рекомендация**: при росте сложности выносить повторяющиеся куски в `entities` (стор/композабл) или в фичу, оставляя в `*.vue` только композицию — но **не ломать** текущий стиль без рефакторинга соседних страниц.

## Auth-страницы

- Общая оболочка **`AuthCard`** из `shared/ui`.
- Сценарии входа — из **`features/auth/...`** (`LoginForm`, поддержка, Keycloak/local).
- На страницах логина подключён **`DebugHeadersForm`** из `@/features/_debug` (отладочный контекст).

## Краткий чеклист для новой страницы

1. Папка **`src/pages/<zone>/<kebab-name>/`**.
2. Компонент **`XxxPage.vue`** (или осознанное исключение по аналогии с auth).
3. **`index.ts`** с `export { default as XxxPage } from './XxxPage.vue'`.
4. Роут в **`app/router-setup`** импортирует из **`@/pages/...`** (без deep-import во внутренности).
5. Зависимости: **widgets / features / shared / entitties**; импорты из **`app/router-setup`** — только если нужны те же типизированные переходы/guards, что и у остальных страниц (и осознавая отклонение от “строгого” FSD).
6. Стили: **scoped SCSS**, БЭМ-классы страницы, при необходимости переменные для `HeaderPage`.
