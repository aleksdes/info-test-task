# Слой `features`

Документ описывает **фактически применённые** в `src/features` паттерны и даёт рекомендации для единообразия нового кода. Базовые определения слоя и направление зависимостей — в скилле `applying-fsd-architecture` и в `src/features/README.md`.

---

## Роль слоя в текущей реализации

В проекте `features` — это **законченные пользовательские сценарии** (кнопки, формы, диалоги, переключатели), которые:

- вызывают методы и сторы **`entities`** (`useRestaurant`, `useEmployees`, `useScheduleTimeline`, `useFaq`, сессия и т.д.);
- при необходимости ходят в API через **`useApiClient()`** и типы из **`@/shared/generated/api.ts`**;
- выносят сценарную логику в **`model/use*`** (композаблы), UI — в **`ui/*.vue`** и сопутствующие типы.

Отдельного сегмента **`api/`** в этом слое нет (как и в `entities`): сетевой слой сосредоточен в `shared` + сгенерированный клиент.

---

## Публичный API слайса

- **`index.ts` в корне слайса** — основная точка входа: реэкспорт Vue-компонента(ов), composable(ов), иногда enum/типов с корня слайса (`switch-table-chart` реэкспортирует `ESwitchTableChartType` из `./types.ts`).
- **`model/index.ts`** — агрегирует `use*.ts` и при необходимости реэкспортирует типы возврата (`Use*Return`, `IUse*Return`).
- **`ui/index.ts`** — реэкспорт компонентов и типов props/emits/slots; часть слайсов дублирует публичные типы в **отдельных `.ts` / `.d.ts`** рядом с фичей (см. ниже).

Импорты снаружи по коду в основном идут **с корня слайса**: `@/features/<slice-name>`.

## Структура сегментов

### `ui/`

- Компоненты в **`PascalCase.vue`**; распространён суффикс **`Feature`** в имени файла и экспорта (`ScheduleCopyDayFeature`, `FaqSearchFeature`).
- Помимо `.vue`, встречаются **`*.ts`** с **контрактами** (props/emits/slots) без разметки (`schedule-copy-day-feature.ts`, `forecast-correction-feature` и т.п.) — типы импортируются в `ui/index.ts` и в `.vue`.
- Рядом лежат **`.d.ts`** (часто `kebab-case.d.ts`) для типизации слотов/props.

### `model/`

- Композаблы **`use<ИмяФичи>`** или **`use<Домен>Feature`** 
- Тип возврата часто оформлен как **`Use*Return`** / **`IUse*Return`** с **JSDoc** на полях интерфейса.
- Связка с доменом: **`storeToRefs`**, **`useApiClient`**, **`useAsyncOperation`**, обёртки над сущностями из `entities`.

### `lib/` (редко)

- Задокументировано в `README` как опциональное;

## Типизация и контракты для компонентов `ui/` сегмента (Vue + TypeScript)

Паттерны по коду:
- Для Vue SFC используются `<script setup lang="ts">`.
- Для пропсов компонентов ui сегмента иногда создаётся отдельный **`*.d.ts`** рядом с компонентом, например `schedule/schedule-feature.d.ts`, и затем тип импортируется в `.vue`.

Рекомендации:
- Для **публичных пропсов/слотов** компонентов ui сегмента держать контракт рядом и стабильно:
- либо `feature-name.types.ts` / `types.ts`,
- либо как сейчас местами сделано — `feature-name.d.ts`.
- На границе модуля/публичного API — **явные типы** (это согласуется с `strict` и правилами TypeScript проекта).

### Прочие папки / файлы на уровне слайса

- **`validation/`** — пример: `schedule-hour-add-row/validation/` с `featureValidation.ts` и `validation/index.ts` (валидация сценария, завязанная на типы фичи).
- **`types.ts`** в корне слайса — enum/типы, не вынесенные в сегмент `types/` (`switch-table-chart/types.ts`).
- **`auth/`** — **нет** корневого `index.ts`; публичные подфичи — отдельные слайсы: `login`, `login-keyclock`, `login-keyclock-local`, каждый со своим `index.ts` (вложенная группировка по домену, не отдельный FSD-слой).

### Сервисные / отладочные слайсы

- **`_debug`** — префикс `_`, экспорт только UI для страниц логина (временная/отладочная фича).
- **`dangerous-drop-cache`** — только `model`, префикс `dangerous` в имени composable (`dangerousUseDropCache`); используется и из другой фичи, и из `app` (см. импорты).

## Именование

- **Директории слайсов**: `kebab-case`; часто суффикс **`-feature`** (`toggle-replacement-aup-feature`), но не везде (`faq-search`, `selector-restaurant`).
- **Компоненты**: `*Feature` или `*FeatureV2`; для кнопок блокировки — уточняющие имена (`LockTimelineButtonFeature` в `toggle-lock-timeline-feature`).
- **Composable**: префикс `use`, суффикс `Feature` там, где это именно сценарий фичи.

## Зависимости и импорты

## Рекомендации для нового кода (выравнивание с существующей базой)

1. Добавлять **корневой `index.ts`** у нового слайса с явным списком экспортов (как у большинства слайсов); для группы логинов — по аналогии с `auth/*` отдельными слайсами без общего barrel, если не нужен единый вход.
2. Сценарную логику с API и состоянием — в **`model/use*.ts`**, тип возврата с **`Use*Return`** и кратким JSDoc на публичных методах.
3. Контракты UI (props/emits/slots) — в **`.ts` рядом с фичей** или **`.d.ts`**, реэкспорт из **`ui/index.ts`**.
4. Публичные enum/типы, не привязанные к одному файлу — по аналогии со **`switch-table-chart/types.ts`** в корне слайса или отдельным файлом, реэкспорт из корневого `index.ts`.
5. Валидацию, тесно связанную с моделью формы фичи — можно держать в **`validation/`** на уровне слайса.
6. Не добавлять **`api/`** в слой `features` без решения по всему проекту — сейчас контракты запросов живут в `shared/generated` и вызовах через `useApiClient`.
7. При новых зависимостях **не повторять** связи `features` → `widgets` / `app`; выносить общее в **`shared`** или поднимать использование на уровень **виджета/страницы**.
