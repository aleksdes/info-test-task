# Слой `entities`

Документ описывает **фактически применённые** в кодовой базе паттерны каталога `src/entities` и даёт рекомендации для единообразия нового кода.

---

## Роль слоя в текущей реализации

В проекте `entities` — это прежде всего:

- **Pinia-хранилища** и типы состояния доменных областей (сотрудники, ресторан, сессия, расписания, статистика и т.д.);
- **композиции `use*`** поверх стора и `@/shared/lib/api-client` (загрузка/мутации, computed-обёртки);
- **точечно — UI-компоненты** (карточки, таблицы админки, медкнижка), экспортируемые из слайса;
- **утилиты трансформации данных** в `lib/` (в т.ч. `lib/data-transformers/`) для таблиц/графиков;
- **вспомогательная логика** в `lib/` у отдельных слайсов (например `session/lib/validateJwt.ts`).

Отдельного сегмента **`api/`** в этом слое **нет**: контракты и вызовы завязаны на сгенерированный клиент `@/shared/generated/api.ts` и вызываются из `model/*`.

---


## Публичный API слайса

- **Корневой `index.ts`** слайса реэкспортирует то, что предназначено для внешнего использования: composable(ы), типы, иногда компоненты из `./ui`, иногда символы из `./lib`.
- **`model/index.ts`** чаще всего агрегирует экспорты из `State.ts`, `use*.ts` и связанных файлов.

---

## Паттерны `model/`

### Pinia

- Почти везде **`defineStore`** с идентификатором в виде строки: `'employees'`, `'restaurant'`, `'shared.filters'`, `'statistics.itph'`, `'list-employees'` и т.д.
- Тип стора часто оформляется как **`I*Store` / `*Store`** и расширяет `Store<'id', ...>` из Pinia.
- Состояние описывается интерфейсами **`I*State`** или **`XState`**.
- Для части сторов включён **`persist: true`** (например `shared-filters`, `admin-roles`) — осознанно учитывать при новых сущностях.

### Композаблы `use*`

- Именование: **`use<Domain>`** (`useRestaurant`, `useEmployees`, `useStatisticsITPH`, …).
- Часто возвращаемый тип явно описан как **`Use*Return`** / **`IUse*Return`** с JSDoc на полях.
- API-вызовы идут через **`useApiClient()`** из `@/shared/lib/api-client`, типы запросов/ответов — из **`@/shared/generated/api.ts`**.
- Расширение доменной модели поверх DTO делается пересечением типов / доп. полями (например `IEmployees` = `EmployeeListModel & { avatar?: … }`) с пометками **`TODO`** там, где сервер ещё не отдаёт поле.

### Файлы состояния

- Распространённые имена: **`State.ts`** (стор + типы), отдельные **`use*.ts`** (логика и fetch).

---

## Паттерны `lib/`

- **`lib/data-transformers/`** — чистые функции приведения данных стора к структурам для различных компонентов, таблиц и графиков (`toTableData`, `toChartData`).
- Зависимости трансформеров: модели из **`*/State.ts`**, типы из **`@/entities/<slice>`** (иногда импорт типа «снаружи» слайса — см. ниже про кросс-импорты).

---

## Паттерны `ui/`

- Компоненты Vue в **`PascalCase.vue`**.
- Рядом лежат **`.d.ts`** с типами props/slots (часто **`kebab-case.d.ts`**), реэкспорт типов из **`ui/index.ts`**.

## Типизация и контракты для компонентов `ui/` сегмента (Vue + TypeScript)

Паттерны по коду:
- Для Vue SFC используются `<script setup lang="ts">`.
- Для пропсов компонентов `ui/` сегмента иногда создаётся отдельный **`*.d.ts`** рядом с компонентом, например `ScheduleItem.d.ts`, и затем тип импортируется в `.vue`.

Рекомендации:
- Для **публичных пропсов/слотов** компонентов `ui/` сегмента держать контракт рядом и стабильно:
- либо `component-name.types.ts` / `types.ts`,
- либо `component-name.d.ts` - **предпочтительно**.
- На границе модуля/публичного API — **явные типы** (это согласуется с `strict` и правилами TypeScript проекта).
  
---

## Зависимости и импорты

### Разрешённое по коду

- **`shared`**: `api-client`, `generated/api`, утилиты дат, типы, UI-примитивы (`calendarV2`, темы).
- **Верхние слои в `entities` не импортируются** (по поиску нет импортов из `@/features/`, `@/widgets/`, `@/pages/`, `@/app/`).
  
---

## Именование и типы

- Префикс **`I`** у интерфейсов доменных сущностей и состояния (`IEmployees`, `IRestaurant`, …).
- Enum и «коды» часто с префиксом **`E`** (`EScheduleForecastRows`, `EStatisticsTypeResources`, `EDateType` из shared).
- Экспорт **дополнительных enum/типов** с корня слайса: пример — **`statistics/types.ts`** с `EStatisticsTypeResources`.

---

## Отсутствие тестов в слое

В **`src/entities`** не найдены каталоги `__unit-tests__` / `__integration-tests__` и типичные `*.spec.*` / `*.test.*`.

**Рекомендация**: при появлении тестов ориентироваться на скиллы проекта (`unit-test-architector`, `integration-test-architector`) и не смешивать сетевые вызовы с юнитами.

---

## Рекомендации для нового кода в `entities`

1. **Сегменты**: по умолчанию **`model/`**; **`ui/`** — только если есть переиспользуемый доменный компонент; **`lib/`** — чистые функции без сайд-эффектов; **`api/`** не вводить отдельно, пока в проекте везде клиент через `shared`.
2. **Публичный API**: собирать в **корневом `index.ts`**, экспортировать типы и функции явным списком (как в существующих слайсах).
3. **Pinia**: сохранять стиль **`defineStore` + `I*State` + `use*`-обёртка** с явным `Use*Return`, если есть сетевые операции.
4. **Импорты между слайсами**: минимизировать; предпочитать **импорт из `index.ts` целевого слайса**; избегать новых deep-import до согласования — либо последовательно рефакторить старые.
5. **DTO**: расширять типы из **`@/shared/generated/api.ts`** на границе `model`, не протаскивать сырой DTO в UI без необходимости.

Этого достаточно, чтобы новые слайсы выглядели **как остальной слой `entities`**, а отклонения от канонического FSD были **осознанными**, а не случайными.
