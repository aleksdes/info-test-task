# FSD Layers — практические пояснения

Этот файл дополняет `SKILL.md` и нужен как рабочая памятка при принятии архитектурных решений.

## Канонический порядок слоёв

Сверху вниз:

`app -> pages -> widgets -> features -> entities -> shared`

Ключевое правило импорта:
- код в слое может импортировать только слои строго ниже;
- обратные зависимости запрещены.

## Важные исключения

- `app` и `shared` не делятся на бизнес-слайсы как обычные слои; в них обычно живут сегменты.
- Внутри `app` и внутри `shared` допустимы свободные импорты между своими сегментами.
- `processes` в актуальной спецификации deprecated: новый код туда не кладём; выносим в `app`/`features`.

## Семантика слоёв (как выбирать слой)

- `shared` — фундамент приложения: инфраструктура, переиспользуемые библиотеки, UI-kit, глобальный config.
- `entities` — бизнес-сущности предметной области (`User`, `Order`, `Product`) и их представление.
- `features` — пользовательские действия/сценарии, которые дают ценность.
- `widgets` — крупные самодостаточные UI-блоки, обычно переиспользуемые или архитектурно изолированные.
- `pages` — экран/маршрут и локальная композиция блоков.
- `app` — корневая инициализация: провайдеры, роутинг, глобальные стили, точка входа.

## Когда НЕ нужно поднимать код выше

Принцип: держим код в самом нижнем возможном слое.

- если код нужен только внутри одной страницы -> оставляем в `pages`, а не выносим в `widgets`;
- если сценарий не переиспользуется -> часто это часть страницы, а не отдельная `feature`;
- если это чистая инфраструктура без домена -> `shared`, а не `entities/features`.

## Сегменты и ответственность

Рекомендуемые сегменты слайса:
- `ui` — визуализация и композиция UI;
- `model` — состояние, use-cases, вычисления, orchestration;
- `api` — запросы, DTO, адаптация внешних данных;
- `lib` — локальные утилиты слайса;
- `config` — локальные флаги/константы.

Правило анти-смешения:
- в `ui` не тянем транспорт/API-детали;
- в `api` не уносим бизнес-оркестрацию;
- DTO не протекают напрямую в UI без необходимости.

## Слайсы и кросс-импорты

- Между слайсами одного слоя импорты по умолчанию запрещены.
- Исключения допустимы только с явным согласованием пользователя.
- Импорт между слоями делаем через публичные API слайсов (barrel `index.ts`), без deep-import, если проект не оговаривает иначе.

## Особенность для связанных сущностей

Если сущности тесно связаны типами, фиксируем связь явно через `@x`-контракты (по FSD), чтобы зависимость была прозрачной и контролируемой.

## Быстрый чек перед завершением задачи

- [ ] Слой выбран по ответственности, а не "где быстрее";
- [ ] Направление зависимостей не нарушено;
- [ ] Нет случайных deep-import во внутренности чужого слайса;
- [ ] `processes` не используется для нового кода;
- [ ] Для `widgets/features` есть оправдание (переиспользование/сценарная ценность).
