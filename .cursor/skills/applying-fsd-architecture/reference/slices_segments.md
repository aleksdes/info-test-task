# Slices и Segments в FSD

Этот файл дополняет `SKILL.md` и фиксирует практические правила по `slices/segments`, чтобы решения были архитектурно стабильными и предсказуемыми.

## 1) Что такое Slice

`Slice` — это группа кода по бизнес-смыслу (продуктовая/доменная область), а не по техническому типу файлов.

- Примеры: `post`, `comments`, `auth`, `product-card`
- Имена слайсов берутся из домена проекта, единого глобального списка нет
- Цель: высокая связность внутри слайса (high cohesion) и нулевая связанность между соседними слайсами слоя (zero coupling)

### Ключевое правило изоляции слайсов

Модуль в слайсе может импортировать:
- код своего слайса;
- код слоёв ниже.

Модуль **не должен** импортировать:
- другие слайсы того же слоя;
- слои выше.

> Исключения допустимы только осознанно и явно зафиксированно в правилах проекта.

## 2) Public API слайса (обязательно)

Каждый слайс должен иметь публичную точку входа (обычно `index.ts`).

Внешний код (вне слайса) импортирует только из public API:

```ts
// good
import { LoginForm } from "features/auth";

// bad (deep import во внутренности слайса)
import { LoginForm } from "features/auth/ui/LoginForm";
```

Зачем это нужно:
- скрывает внутреннюю структуру;
- делает рефакторинг безопаснее;
- удерживает архитектурные границы на уровне импортов.

## 3) Slice groups (группы слайсов)

Допускается группировать близкие слайсы в папку:

```text
features/
  post/
  comment/
  moderation/
```

Важно: группа — это только структурная папка, **не место для общего кода**.  
Общий код выносится в отдельный корректный слайс/слой (обычно `shared` или отдельный `entity/feature` по смыслу).

## 4) Что такое Segment

`Segment` — техническая декомпозиция внутри слайса (последний уровень иерархии).

Стандартные сегменты:
- `ui` — компоненты, стили, форматтеры для отображения;
- `api` — запросы, DTO, мапперы транспорта;
- `model` — бизнес-логика, стейт, схемы;
- `lib` — внутренние утилиты слайса;
- `config` — флаги и конфиг слайса.

Сегменты нужны для читаемости и предсказуемой навигации, но не должны ломать бизнес-границы слайса.

## 5) Именование сегментов

Имя сегмента должно отражать **назначение**, а не тип сущностей.

- good: `ui`, `model`, `api`, `config`
- bad: `components`, `hooks`, `types`

Почему `components/hooks/types` плохие:
- они описывают форму кода, но не его роль;
- по ним сложнее быстро понять, где лежит сценарий/ответственность.

## 6) Особые случаи App и Shared

Слои `app` и `shared` обычно не делятся на бизнес-слайсы, поэтому для них особенно важны аккуратные сегменты и их public API.

- В `shared` не размещаем бизнес-логику;
- В `app` держим глобальную композицию/инициализацию приложения;
- Для сегментов в этих слоях действует то же правило: наружу только через public API.

## 7) Практический чеклист перед изменениями

Перед добавлением/перемещением кода проверь:

1. Код лежит в минимально допустимом слое.
2. Бизнес-смысл изолирован в одном слайсе.
3. Нет deep-import из чужого слайса.
4. У слайса/сегмента есть явный public API.
5. Сегмент назван по назначению, а не по форме файлов.
6. Общий код не прячется в slice-group папке.

## 8) Мини-шаблон структуры слайса

```text
features/
  auth/
    api/
    model/
    ui/
    lib/
    config/
    index.ts
```

`index.ts` экспортирует только внешний контракт слайса. Внутренние файлы остаются приватными для слайса.
