# Widgets directory (`src/widgets`) — правила и паттерны проекта

Этот документ **не про теорию FSD**, а про **локальные конвенции**, которые уже реально применены в коде `src/widgets/`. Цель — чтобы будущие виджеты выглядели и собирались так же.

## Роль слоя `widgets`

В текущей кодовой базе `widgets` — это:
- **крупные переиспользуемые UI-блоки**, которые часто соответствуют “большому куску страницы” (пример: `schedule`, `schedule-statistics`, `statistics/*`).
- **композиция нескольких `features` + `entities` + `shared/ui`** в единый блок.
- иногда **layout-обёртки/скеффолды** (см. `widgets/_layout`).

(Расхождение/риск: часть виджетов тянет зависимости из `app/*` — это импорт “вверх” по слоям и противоречит направлению `app -> pages -> widgets -> ...`.)

## Именование и структура директорий

- **Имена слайсов виджетов**: в проекте встречаются в основном `kebab-case` (`schedule-statistics`, `schedule-forecast`, `network-accessibility-warning`, `users-schedule-calendar`, `tu-aup-schedule-calendar`, `aup-schedule-calendar`)
  Рекомендация:
  - для новых виджетов держать **слайс виджета в `kebab-case`**;
  - если нужна группировка — использовать **папку-домен** (как `statistics/`) и внутри неё всё равно держать `kebab-case` 

- **Ожидаемые сегменты внутри виджета** (если применимо):
  - `index.ts` — **public API** виджета (реэкспорт “точки входа”).
  - `ui/` — внутренние компоненты/части виджета, допускается иерархия (`ui/schedule-day/*`, `ui/schedule-week/*`).
  - `model/` — состояние/сценарии/трансформации данных, часто с `model/index.ts` как баррелем.


## Public API (`index.ts`) — как оформлять

Паттерн проекта: `index.ts` почти всегда **реэкспортирует один “главный” компонент**.

Примеры:
- `widgets/schedule/index.ts` → `ScheduleWidgetV2`
- `widgets/schedule-statistics/index.ts` → `StatisticsWidget`
- `widgets/_layout/index.ts` → несколько scaffold-компонентов

Рекомендации:
- **Один основной экспорт** на виджет-слайс (если не layout-агрегатор).
- Имя экспорта — `PascalCase` + суффикс `Widget`
- Внутреннее дерево `ui/*` и `model/*` наружу **не торчит**
- Если наружу нужны типы — экспортировать их из **сегментного `ui/index.ts` или `model/index.ts`**, а затем (опционально) пробросить из корневого `index.ts` виджета.

## Типизация и контракты (Vue + TypeScript)

Паттерны по коду:
- Для Vue SFC используются `<script setup lang="ts">`.
- Для пропсов виджетов иногда создаётся отдельный **`*.d.ts`** рядом с компонентом, например `schedule/schedule-widgetV2.d.ts`, и затем тип импортируется в `.vue`.

Рекомендации:
- Для **публичных пропсов/слотов** виджета держать контракт рядом и стабильно:
- либо `widget-name.types.ts` / `types.ts`,
- либо `widget-name.d.ts`. - **предпочтительно**.
- На границе модуля/публичного API — **явные типы** (это согласуется с `strict` и правилами TypeScript проекта).

Эти правила касаются и всех других `ui` компонентов `*.vue` слоя

## Зависимости и импорты (важно для FSD)

### Разрешённые зависимости (как должно быть)
Для виджета допустимо импортировать:
- `shared/*` (ui/lib/config)
- `entities/*`
- `features/*`

И использоваться виджет должен из:
- `pages/*` (основной потребитель),
- `app/*` (агрегация приложения).

## Чеклист для нового виджета (в стиле этого проекта)

- **Структура**: `index.ts` + `<WidgetName>.vue` + (по необходимости) `ui/`, `model/`.
- **Public API**: 1 основной export, без deep-import наружу.
- **Типы**: явные типы пропсов на границе; при необходимости отдельный файл `*.d.ts`/`types.ts`.
- **Зависимости**: только вниз по слоям (`shared/entities/features`), без `app/*` внутри.
- **Побочки**: по возможности в `model/`/`features`, не в `.vue` с прямым транспортом.
