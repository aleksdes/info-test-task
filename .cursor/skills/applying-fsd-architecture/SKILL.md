# Применение архитектуры FSD

### Когда применять
Этот навык автоматически активируется в следующих ситуациях:

- Вопросы, связанные со структурой проекта или с архитектурой FSD (слои, слайсы, сегменты)
- Проектирование структуры папок проекта
- Вопросы о правилах зависимостей и направлении импортов
- Использование слоёв shared, entities, features, widgets, pages и других
- Миграция или рефакторинг на FSD

### Trigger Patterns
- Упоминание «FSD», «Feature-Sliced Design»
- «Структура слоёв», «создание слайса», «добавление сегмента»
- «shared слой», «entities слой», «features слой» и т.д.
- «Правила зависимостей», «направление импорта»
- «Архитектура фронтенда», «проектирование структуры папок»

### Базовые проверки:
- Соблюдай направление слоев: `app -> pages -> widgets -> features -> entities -> shared`
- Избегай кросс-импортов между слайсами одного слоя
- Разделяй ответственности по сегментам (`ui`, `model`, `api`, `lib`, `config`)
- Старайся размещать код в самом нижнем допустимом слое
- Помни исключения: `app` и `shared` — слои без бизнес-слайсов, внутри них импорты между сегментами допустимы
- Не используй `processes` для нового кода (слой deprecated)

### ВАЖНО: особенности и паттерны этого проекта (обязательные reference-доки)

## Процесс

1. Классифицируй запрошенное изменение
   - Новая бизнес-возможность -> обычно `features`
   - Доменный объект/состояние -> обычно `entities`
   - Переиспользуемый примитив/ui/helper -> обычно `shared`
   - Композиция на уровне маршрута -> обычно `pages`
   - Крупный самодостаточный переиспользуемый UI-блок -> `widgets`
   - Каркас приложения/провайдеры/bootstrap -> обычно `app`

2. Проверь направление зависимостей
   - Верхние слои могут импортировать нижние
   - Нижние слои не должны импортировать верхние
   - Импорты slice-to-slice внутри одного слоя не допускаются (кроме явно согласованных исключений)
   - Для `app`/`shared` применяй правило сегментов, а не бизнес-слайсов

3. Спроектируй структуру папок
   - Создавай или обновляй только нужные сегменты
   - Поддерживай единообразное именование сегментов
   - Держи публичные точки входа явными
   - Для `entities` с тесными связями типов используй `@x`-контракты

4. Реализуй и проверь
   - Обнови импорты так, чтобы границы оставались валидными
   - Запусти линт/type-проверки, если доступны
   - Зафиксируй любые неизбежные компромиссы

При возврате архитектурных решений используй:

```markdown
## Размещение в FSD
- Слой: <выбранный слой>
- Слайс: <имя слайса>
- Сегменты: <ui/model/api/lib/config>

## Валидация зависимостей
- Разрешенные импорты: <список>
- Запрещенные импорты устранены/избежаны: <список>

## Внесенные изменения
- <изменение файла/папки 1>
- <изменение файла/папки 2>

- <ограничения, edge cases, заметки по миграции>
```

## Частое соответствие

- Обертки API-клиента, утилиты дат/форматирования, атомы дизайн-системы -> `shared`
- Бизнес-сущности (`User`, `Product` и т.д.) models/api/ui -> `entities/<entity>`
- Пользовательские действия с продуктовой ценностью (auth, like, шаг checkout) -> `features/<feature>`
- Крупные композитные блоки, переиспользуемые между страницами -> `widgets/<widget>`
- Страницы уровня маршрутов и локальная композиция -> `pages/<page>`

## Антипаттерны, которых следует избегать
❌ Перекрестный импорт между фрагментами на одном слое

```js
// features/comments/ui/CommentList.tsx
import { likePost } from '@/features/like-post'; // BAD!
```
✅ Решение : Используйте обозначение @x для сущностей или компонуйте на более высоком уровне.
```js
// For entities with business relationships:
// entities/user/@x/order.ts
export { UserOrderHistory } from './ui/UserOrderHistory';

// For features, compose at page level:
// pages/post/ui/PostPage.tsx
import { CommentList } from '@/features/comments';
import { LikeButton } from '@/features/like-post';
```
❌ Бизнес-логика в Shared
```js
// shared/lib/userHelpers.ts
export const calculateUserReputation = (user) => { ... }; // BAD!
```
✅ Решение : Переместить на слой объектов
```js
// entities/user/lib/calculateReputation.ts
export const calculateUserReputation = (user) => { ... };
```
❌ Обход публичного API
```js
import { LoginButton } from '@/features/auth/ui/LoginButton'; // BAD!
```
✅ Используйте общедоступный API
```js
import { LoginButton } from '@/features/auth'; // GOOD!
```
❌ Бог берет на себя слишком большую ответственность (излишнюю ответственность)
```js
// features/user-management/  ← TOO BROAD
//   - login, register, profile-edit, password-reset, etc.
```
✅ Разделено на сфокусированные функции
```js
// features/auth/
// features/profile-edit/
// features/password-reset/
```
