# Unit-test architector (FSD)

## Когда применять

Пользователь просит unit-тесты, покрытие, моки API, структуру тестов в слайсе — или правит код в `entities` / `features` / `widgets` / `pages` / `app` / `shared`.

Согласованность с архитектурой: [applying-fsd-architecture](../applying-fsd-architecture/SKILL.md).

## Обязательные шаги перед кодом

1. **Стек тестов** — взять из корня целевого приложения: `package.json` (и при наличии `vitest.config.*`, `jest.config.*`, `vite.config.*` с тестовым блоком). Раннер и библиотеки ассертов/моков — **только те, что уже в проекте** (или явно добавить зависимости и зафиксировать в `package.json`). Если в workspace только правила — опираться на `package.json` репозитория приложения, который пользователь указывает или где лежит код.
2. **Размещение** — все unit-файлы только в **`__unit-tests__`**, на одном уровне с тестируемым функционалом внутри **того же слайса/той же области слоя** (см. ниже).
3. **Сеть** — запрещены реальные HTTP-запросы к боевым/stage серверам. Мокать клиент, `fetch`, адаптеры в `api/`, ответы DTO. Паттерны: [references/mocking.md](references/mocking.md).
4. **Покрытие** — ориентир **не выше 90%** по строкам/веткам для unit-слоя: критичная логика и регрессии, без гонки за 100% и без тестирования тривиального шума. Детали: [references/coverage.md](references/coverage.md).

## Куда класть `__unit-tests__`

| Область | Путь (пример) |
|--------|----------------|
| Сущность | `src/entities/<slice>/__unit-tests__/` |
| Фича | `src/features/<slice>/__unit-tests__/` |
| Виджет | `src/widgets/<slice>/__unit-tests__/` |
| Страница | `src/pages/<slice>/__unit-tests__/` |
| App | `src/app/__unit-tests__/` (или `app/__unit-tests__/` по структуре репо) |
| Shared | `src/shared/<segment-or-subfolder>/__unit-tests__/` — рядом с тестируемым модулем, не «общая свалка» на весь shared |

Импорты в тестах — через **public API** слайса, как в прод-коде; внутри слайса допустимы deep-import по правилам проекта.

## Что тестировать в первую очередь

- `model/`, `lib/` — сценарии, чистые функции, редьюсеры/селекторы.
- `api/` — разбор/маппинг ответов при **подставленных** мок-ответах, не живой сетью.
- `ui/` — по стеку проекта (Testing Library / happy-dom / jsdom); без реального бэкенда.

Подробные сценарии и примеры структуры файлов: [references/placement-examples.md](references/placement-examples.md).
