# Моки сети и API в unit-тестах

## Принципы

- Unit-тест не должен зависеть от доступности сервера, DNS, TLS, латентности.
- Мокается **граница I/O**: HTTP-клиент (axios/fetch-обёртка), функции `api/`-слоя, иногда глобальный `fetch` — в зависимости от того, что реально вызывается кодом под тестом.

## Типовые приёмы (подставить API раннера проекта)

### Vitest

- `vi.mock('module')`, `vi.spyOn`, `vi.stubGlobal('fetch', fn)`.
- MSW — если уже в зависимостях: перехват на уровне `fetch`/XHR в тестах, без реального хоста (всё равно без боевого URL).

### Jest

- `jest.mock`, `jest.spyOn`, подмена `global.fetch` в `beforeEach` / `afterEach`.

## Что мокать

| Слой | Что подменять |
|------|----------------|
| `api/` | Транспорт возвращает фиктивный `Response` или resolved value клиента; проверять маппинг в доменную модель. |
| `model/` | Репозиторий / `getXxx` как функции-заглушки или мок-модуль с фикстурами DTO. |
| Хуки с запросами | Обёртка данных (React Query и т.д.) — провайдеры тестовой оболочки + мок `queryFn` / начальный кэш, без сети. |

## Антипаттерны

- «Проверочный» запрос на staging/production.
- Случайные порты и реальные сервера в CI без изоляции (это уже интеграционный уровень, не unit).
