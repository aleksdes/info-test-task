# Моки сети и API в интеграционных тестах

## Принципы

- Интеграционный тест проверяет **сборку нескольких частей**, но **не** доступность внешнего сервера, DNS, TLS, сетевую латентность.
- Подменяется **граница I/O**: HTTP-клиент, модуль `api/`, при необходимости глобальный `fetch` — в соответствии с тем, что реально вызывается под тестом.
- Реальные URL боевого/stage API в тестах **не использовать**.

## Vitest

- `vi.mock('module')`, `vi.spyOn`, `vi.stubGlobal('fetch', fn)`.
- MSW (`setupServer` / `http.get` и т.д.) — если пакет уже в зависимостях: перехват в Node/jsdom без исходящего запроса на хост (handlers возвращают фикстуры).

## Jest

- `jest.mock`, `jest.spyOn`, подмена `global.fetch` в `beforeEach` / `afterEach`.
- MSW — аналогично, при наличии в проекте.

## React Query / аналоги

- Тестовый `QueryClient` с короткими `retry: false`, `gcTime: 0` где уместно.
- `queryFn` или клиент — моки; начальное состояние кэша — только из фикстур, не из сети.

## Что проверять при моках

| Слой | Фокус |
|------|--------|
| `api/` | Согласованность маппинга DTO → домен при **подставленных** ответах. |
| `model/` + `ui/` | Сценарий от действия пользователя до состояния/запроса — без живого бэкенда. |

## Антипаттерны

- Запросы к реальному API в CI или локально «для проверки».
- Неизолированный локальный сервер без явного решения команды (это уже другой уровень тестов, не описанный этим скиллом).
