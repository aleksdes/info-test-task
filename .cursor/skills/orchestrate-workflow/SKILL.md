# Оркестрация субагентов (workflow)

## Когда применять

Применяй этот скилл, когда пользователь приносит **большую комплексную задачу**, которая:

- затрагивает много файлов/модулей/слоёв;
- требует декомпозиции на подзадачи;
- включает риск архитектурных нарушений (FSD, публичные API, импорты);
- потенциально требует UI/UX соответствия макетам;
- обязательно должна завершиться тестами и прогоном тестов.

## Trigger patterns

- «сделай фичу целиком», «большая задача», «рефакторинг + добавление функционала»
- «нужно спланировать», «декомпозируй», «сделай по шагам»
- «после реализации проверь архитектуру/FSD»
- «есть макеты Figma», «сверь с дизайном/макетом»
- «добавь unit/integration тесты и прогони их»

## Обязательные входы / допущения

- **Большая задача** трактуется как “нужно планирование” → всегда стартуем с `agent-planner`.
- Планер создаёт файлы в `tasks-planner/<ветка|дата>/<понятное краткое имя задачи>` и `tasks.md` — это **единая точка правды** по списку подзадач.
- Подзадача считается “выполнена”, когда:
  - реализация завершена,
  - ревью не находит блокеров,
  - статус подзадачи отмечен ✅ в `tasks.md` (или эквивалентном чекере).

## Перед реализацией **Важно**
Если проект уже существует и в нем определен стек технологий в config файлах или package файлах, сначала определи их и используй. НЕ НУЖНО переписывать стек технологий если это явно не задано пользователем. НЕ НУЖНО переписывать код на другой стек. НЕ НУЖНО выдумывать причины перехода на другой стек. Если в ходе разработки потребуется добавить библиотеку или пакет, они не должны требовать смены стека, должны быть совместимы с текущим стеком технологий в проекте.    

## Субагенты, которые обязаны участвовать

1. `@.cursor/agents/agent-planner.md`
2. `@.cursor/agents/agent-worker.md` - должен запускаться при любых обстоятельствах. Задачи не должны выполняться основным контекстом. Если agent-worker не может стартовать нужно остановить полностью весь процесс и спросить у пользователя попытку перезапустить agent-worker. Если и это не помогает - полностью остановить процесс и выдать Понятную ошибку с причиной отмены всего процесса
3. `@.cursor/agents/agent-code-reviewer.md`
4. `@.cursor/agents/design-iterator.md` (условно, только если есть дизайн-ресурсы)
5. `@.cursor/agents/unit-test-creator.md`
6. `@.cursor/agents/integration-test-creator.md`
7. `@.cursor/agents/test-runner.md`

## Процесс (строго по шагам)

### Шаг 1 — запуск планера (обязателен)

Сразу после поступления комплексной задачи:

- запусти субагента `agent-planner`;
- дождись, пока он:
  - создаст/найдёт `tasks-planner/<ветка|дата>/<понятное краткое имя задачи>`,
  - заполнит `tasks.md`,
  - создаст `.md` для каждой подзадачи.

Если план уже существует и была дана задача продолжить выполнение плана (папка по ветке уже есть) — планер **ничего не переписывает** (см. правила планера), используем существующий план.

### Шаг 2 — итеративная реализация подзадач (worker + reviewer)
этот шаг должен запускаться при любых обстоятельствах. Задачи не должны выполняться основным контекстом. Если agent-worker не может стартовать нужно остановить полностью весь процесс и спросить у пользователя попытку перезапустить agent-worker с момента сбоя. Если и это не помогает - полностью остановить процесс и выдать Понятную ошибку с причиной отмены всего процесса

Дальше работаем циклами по подзадачам из `tasks.md`.

Алгоритм цикла:

1) Выбери **следующую** подзадачу со статусом ❌.

2) Запусти субагента `agent-worker` для реализации **этой** подзадачи и всех её подзадач/шагов из файла подзадачи.

3) После завершения воркера **обязательно** запусти субагента `agent-code-reviewer`, передав ему:
   - diff (предпочтительно) или список изменённых файлов,
   - цель подзадачи (1–2 предложения),
   - любые ошибки линтера/типов (если появились).

4) Если субагент `agent-code-reviewer` нашёл **блокеры (must-fix)**:
   - возвращайся к пункту 2) для этой же подзадачи (новая итерация воркера),
   - снова запускай ревьюера,
   - повторяй, пока блокеры не исчезнут.

5) Когда блокеров нет и подзадача действительно выполнена — отметь её как ✅ в `tasks.md` (или оставь инструкцию, что статус должен быть обновлён в план-файлах, если это делает отдельный агент/процесс в вашем репозитории).

Остановка цикла: когда **все** подзадачи в `tasks.md` стали ✅.

### Шаг 3 — дизайн-итерации (условно)

Этот шаг выполняется **только если** в задаче/контексте были предоставлены:

- макеты/ссылки/фреймы Figma,
- или любые другие ресурсы, задающие внешний вид и UI/UX требования (скриншоты, спецификации, токены, гайдлайны).

Если дизайн-ресурсов **не было** — шаг пропускается полностью.

Если дизайн-ресурсы были:

- запусти субагента `design-iterator` **после** того, как завершены шаги реализации и ревью по всем подзадачам;
- цель: найти несоответствия реализации макетам и исправить их **только если** это:
  - не ломает поведение/контракты,
  - не ухудшает архитектурные границы (FSD) и не требует рискованных рефакторингов.

Все правки должны выполняться с учетом `@.cursor/skills/applying-fsd-architecture/SKILL.md`

Если для исправления дизайна нужна потенциально рискованная правка логики/архитектуры — фиксируй расхождение и выноси на решение пользователя.

### Шаг 4 — тесты (сначала unit, потом integration)

После завершения реализации (и после дизайн-итераций, если они были):

1) Запусти субагента `unit-test-creator` и добавь необходимые unit-тесты (строго по скиллу `unit-test-architector`).
2) Затем запусти субагента `integration-test-creator` и добавь необходимые integration-тесты (строго по скиллу `integration-test-architector`).

Ограничения:

- никаких реальных сетевых запросов;
- импорты через public API слайсов;
- покрытие ориентировать на критичную логику/флоу, без фанатизма (>90% не цель).

### Шаг 5 — прогон всех тестов в проекте (test-runner)

После добавления тестов:

- запусти субагента `test-runner` **для всех тестов проекта**, 
  - для этого репозитория это означает минимум: `npm run test:unit`
  - и дополнительно (если есть/нужны интеграционные проверки UI/контрактов): `npm run test:integration`

Если тесты **не прошли**:

- выдай пользователю **структурированный отчёт**:
  - какие файлы/наборы тестов упали,
  - названия упавших тестов/кейсов,
  - сообщения ошибок/стектрейсы (кратко, по сути),
  - предположительная причина (если очевидно) и варианты: “чинить тесты” vs “чинить код”.

Критическое правило:

- **любые правки после test-runner** (и тестов, и прод-кода) — **только после явного согласования пользователя**.

### Шаг 6 — финальная сверка и проставление статусов плана (обязательно)

После выполнения **всех** задач (и дизайн-итераций, если они были) необходимо:

- пройтись по `tasks-planner/<ветка|дата>/<понятное краткое имя задачи>/tasks.md`;
- **проверить по факту**, какие подзадачи действительно выполнены, а какие нет;
- **проставить статусы** ✅/❌ для каждой подзадачи (и глобальной задачи, если чекер присутствует).
  
## Формат результата для пользователя (чтобы было легко контролировать)

После выполнения workflow верни:

- **Статус**: все подзадачи ✅/есть ❌ (с указанием каких).
- **Ключевые изменения**: 3–10 пунктов по сути.
- **Тесты**: какие unit/integration добавлены и где лежат.
- **Прогон**:
  - если зелёный — что именно прогонялось,
  - если красный — структурированный отчёт падений (см. шаг 5) и вопрос “что фиксируем”.
