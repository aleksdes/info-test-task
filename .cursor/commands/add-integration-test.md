**перед работой ВАЖНО**
По умолчанию **не изменяй** файлы `.cursor/**` (skills/rules/agents/commands): они считаются “инфраструктурой промтов”.
Исключение: если пользователь **явно** попросил улучшить/исправить ИИ‑инфраструктуру — изменения допустимы, но их нужно перечислить в результате.

Ты — команда **add-integration-test** для этого репозитория.

Твоя задача: **добавить integration-тест(ы)** для указанной пользователем области кода (директория/слайс/модуль), соблюдая FSD, TypeScript strict и текущий тестовый стек проекта.

Обязательный источник правил:
- `@.cursor/skills/integration-test-architector/SKILL.md`

Если в процессе затрагиваются границы слоёв/публичные API/размещение по FSD — дополнительно применять:
- `@.cursor/skills/applying-fsd-architecture/SKILL.md`

## Вход

Пользователь должен передать:
- путь к целевой директории/слайсу/модулю, где нужно добавить тесты (пример: `src/features/auth` или `src/widgets/header`),
- (опционально) конкретный флоу/сценарий, который нужно покрыть (лучше в виде шагов),
- (опционально) список файлов/функций/компонентов, которые участвуют,
- (опционально) баг/регрессия (что сломано и ожидаемое поведение).

Если пользователь указал только директорию — выбери **минимально затронутый** и **наиболее критичный** для регрессий флоу внутри неё и зафиксируй допущение.

## Обязательный процесс (не пропускать)

1) Сразу прочитай и выполни `@.cursor/skills/integration-test-architector/SKILL.md`.
2) Определи тестовый стек из `package.json` (и при наличии `vitest.config.*`/`jest.config.*`/`playwright.config.*`/`vite.config.*`/ другого ранера тетсов):
   - используй **только** уже установленный раннер/библиотеки;
   - не добавляй зависимости без явной необходимости задачи.
3) Размещение:
   - клади integration-тесты **только** в `__integration-tests__` рядом с тестируемой областью внутри **того же** слоя/слайса (co-location), как требует скилл. Также поступай с `__integrations-tests__` и `__e2e-tests__`
4) Сеть запрещена:
   - никаких реальных HTTP-запросов (включая stage);
   - мокай `fetch`/клиент/адаптеры/слой `api/`, DTO-ответы; при необходимости используй MSW **только** в тестовой среде и без боевого хоста (если он уже есть в проекте).
5) Импорты:
   - в тестах импортируй зависимости через **public API** слайсов так же, как в прод-коде;
   - deep-import допускается только в рамках правил проекта (в первую очередь внутри того же слайса).
6) Покрытие:
   - покрывай критичные сквозные сценарии и контракты, **не гонись за 100%**, ориентир — **не выше 90%** для интеграционного набора.
7) После добавления:
   - убедись, что тесты компилируются и проходят в рамках доступных инструментов (если можешь запустить — запускай только новые/изменённые тесты).

## Формат ответа

Верни:
- какие тесты добавлены и где лежат (группируя по директории `.../__integration-tests__/...`),
- что именно замокано (fetch/клиент/адаптер/слой `api/`/DTO/MSW),
- какой сценарий/контракт покрыт и какие риски остались (кратко).
