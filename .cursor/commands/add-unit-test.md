**перед работой ВАЖНО**
По умолчанию **не изменяй** файлы `.cursor/**` (skills/rules/agents/commands): они считаются “инфраструктурой промтов”.
Исключение: если пользователь **явно** попросил улучшить/исправить ИИ‑инфраструктуру — изменения допустимы, но их нужно перечислить в результате.

Ты — команда **add-unit-test** для этого репозитория.

Твоя задача: **добавить unit-тест(ы)** для указанной пользователем области кода (директория/слайс/модуль), соблюдая FSD, TypeScript strict и текущий тестовый стек проекта.

Обязательный источник правил:
- `@.cursor/skills/unit-test-architector/SKILL.md`

Если в процессе затрагиваются границы слоёв/публичные API/размещение по FSD — дополнительно применять:
- `@.cursor/skills/applying-fsd-architecture/SKILL.md`

## Вход

Пользователь должен передать:
- путь к целевой директории/слайсу/модулю, где нужно добавить тесты (пример: `src/features/auth` или `src/shared/lib/date`),
- (опционально) список файлов/функций/компонентов, которые тестировать,
- (опционально) баг/регрессия (что сломано и ожидаемое поведение).

Если пользователь указал только директорию — выбери **минимально затронутый** и **наиболее критичный** для регрессий модуль внутри неё и зафиксируй допущение.

## Обязательный процесс (не пропускать)

1) Сразу прочитай и выполни `@.cursor/skills/unit-test-architector/SKILL.md`.
2) Определи тестовый стек из `package.json` (и при наличии `vitest.config.*`/`jest.config.*`/`vite.config.*`/ другого ранера тестов):
   - используй **только** уже установленный раннер/библиотеки;
   - не добавляй зависимости без явной необходимости задачи.
3) Размещение:
   - клади unit-тесты **только** в `__unit-tests__` рядом с тестируемым кодом внутри **того же** слайса/области слоя. Также поступай с `__integrations-tests__` и `__e2e-tests__`
4) Сеть запрещена:
   - никаких реальных HTTP-запросов; мокай транспорт (`fetch`/клиент/адаптеры) и DTO/ответы.
5) Импорты:
   - в тестах импортируй зависимости через **public API** слайсов так же, как в прод-коде;
   - deep-import допускается только в рамках правил проекта (в первую очередь внутри того же слайса).
6) Покрытие:
   - целиться в критичную логику/регрессии, **не гнаться за 100%**, ориентир — до 90%.
7) После добавления:
   - убедись, что тесты компилируются и проходят в рамках доступных инструментов (если можешь запустить — запусти только новые/изменённые тесты).

## Формат ответа

Верни:
- какие тесты добавлены и где лежат (группируя по директории `.../__unit-tests__/...` и другие типы тестов),
- что именно замокано (fetch/клиент/адаптер/DTO),
- какие кейсы покрыты и какие риски остались (кратко).
