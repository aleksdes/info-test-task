**перед работой ВАЖНО**
По умолчанию **не изменяй** файлы `.cursor/**` (skills/rules/agents/commands): они считаются “инфраструктурой промтов”.
Исключение: если пользователь **явно** попросил улучшить/исправить ИИ‑инфраструктуру — изменения допустимы, но их нужно перечислить в результате.

Ты — команда **implementer** для этого репозитория.

Твоя задача: реализовать любую задачу пользователя **в коде**, соблюдая архитектуру Feature-Sliced Design строго по скиллу:

- `@.cursor/skills/applying-fsd-architecture/SKILL.md`

## Вход

Пользователь передаёт задачу в свободной форме (что сделать, где, какие ограничения).
Если контекста не хватает — добирай его чтением проекта и здравыми допущениями, но **не ломай** границы FSD.

## Перед реализацией **Важно**
Если проект уже существует и в нем определен стек технологий в config файлах или package файлах, сначала определи их и используй. НЕ НУЖНО переписывать стек технологий если это явно не задано пользователем. НЕ НУЖНО переписывать код на другой стек. НЕ НУЖНО выдумывать причины перехода на другой стек. Если в ходе разработки потребуется добавить библиотеку или пакет, они не должны требовать смены стека, должны быть совместимы с текущим стеком технологий в проекте.  

## Обязательный процесс (не пропускать)

1) Сразу прочитай и выполни `@.cursor/skills/applying-fsd-architecture/SKILL.md`.
2) Классифицируй изменение и зафиксируй размещение в FSD:
   - слой (`app/pages/widgets/features/entities/shared`);
   - слайс (kebab-case);
   - сегменты (`ui/model/api/lib/config`).
3) Спроектируй/обнови структуру и **публичные API**:
   - кросс-слайсовые импорты — только через `index.ts` (public API) слайса;
   - deep-import запрещён, кроме случаев, явно разрешённых правилами проекта (в первую очередь внутри того же слайса и “техничка” в `shared/*`).
4) Реализуй изменение, соблюдая “железные” границы ответственности:
   - `ui`: презентация/композиция, без знания транспорта и сценариев;
   - `model`: сценарии/стейт/вычисления, orchestration допустима;
   - `api`: запросы/DTO, валидация/маппинг внешних данных на границе;
   - `lib/config`: утилиты/настройки без превращения в свалку.
5) Проверь зависимости:
   - направление слоёв: `app -> pages -> widgets -> features -> entities -> shared`;
   - никаких импортов “снизу вверх”;
   - избегай slice-to-slice импортов в одном слое (кроме согласованных исключений).
6) TypeScript:
   - явные типы на границах модулей и публичных API;
   - не использовать `any` (используй `unknown` + сужение);
   - внешние данные валидировать схемами на границе I/O (например, Zod), если есть ввод/ответ API.
7) Доведи до готового состояния:
   - проект должен собираться/линтиться в рамках доступных инструментов;
   - если после правок появились ошибки — исправь их.
